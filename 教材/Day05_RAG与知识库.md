## 1. 小白先懂：为什么模型会“胡说”

LLM 的本质是“根据上下文预测下一个词”。

当 PRD 不完整、或者业务知识不在上下文里时，它会：

- 编造细节
- 混淆概念
- 给出看似合理但不符合你业务的答案

RAG（Retrieval-Augmented Generation）要解决的问题是：

> 让模型生成前先查资料，把相关内容放进上下文，再让它基于资料生成。

新手理解版：

> 你把“参考资料”塞给模型，它就更像“开卷考试”，不容易瞎编。

## 3. 向量检索到底在干什么（用人话讲）

### 3.1 核心流程：分块 → 向量化 → 存储 → 检索

1) 分块（chunking）

- 把长文档切成一段一段
- 每段有一个 `chunk_id`

2) 向量化（embedding）

- 把每段文字变成一串数字（向量）
- 相似的文字向量距离更近

3) 存储（vector store）

- 把向量+原文存到向量库（这里是 ChromaDB）

4) 检索（retrieval）

- 你给一个查询（比如“登录失败场景”）
- 向量库返回最相似的 top_k 段原文

你要记住一句话：

> RAG 的关键不是“让模型更聪明”，而是“让模型有依据”。

再补一句工程化的：

> RAG 的质量主要取决于：分块是否合理 + embedding 是否合适 + 检索 top_k 是否匹配业务。

## 4. 分块策略怎么设计（这就是工程能力）

你会在 `rag_service.py` 里看到 chunking 相关逻辑。

分块时你要考虑：

- chunk 太大：检索回来的片段太长，占上下文，成本高
- chunk 太小：信息被切碎，检索命中但上下文不完整

常见策略：

- 固定长度（按字符数/词数）
- 带重叠 overlap（避免语义切断）
- 按标题/段落结构切（更贴近文档语义）

面试加分说法：

> “我会根据 PRD 的结构，优先按章节/标题切分，并设置适当 overlap，兼顾语义完整与召回精度。”

### 4.1 手把手：在本项目里找到 chunk 参数在哪里配置

打开：`backend/app/config.py`，你会看到类似配置：

- `CHUNK_SIZE`
- `CHUNK_OVERLAP`
- `RAG_TOP_K`

这 3 个就是 RAG 的“旋钮”。

新手先记一个默认理解：

- chunk_size 越大：上下文更完整，但更占 token
- overlap 越大：更不容易切断语义，但存储与计算更大
- top_k 越大：参考资料更多，但更可能掺杂噪音

## 5. 知识库业务流程怎么串起来

你可以把知识库理解为“可选的参考资料集合”。

典型流程（对应 `knowledge_base_service.py`）：

1) 创建知识库（collection）
2) 上传文档
3) 解析成文本
4) 分块 + embedding
5) 写入 ChromaDB

删除流程：

- 删除某个文档的所有 chunks
- 删除整个 collection

为什么要支持删除？

- 资料会过期
- 有些资料可能敏感
- 需要让用户“可控”

## 6. 从 0 开始写一个“最小 RAG”（不接 UI，只做检索体验）

你做完这个最小例子，就会懂向量库到底在干什么。

### 6.1 目标

- 你有 3 段文本
- 你输入一个问题
- 系统返回最相关的那段文本

### 6.2 伪代码（概念版）

```python
texts = ["登录成功...", "登录失败...", "注册流程..."]
embeddings = embed(texts)
store(embeddings, texts)

q = "登录失败怎么办"
q_vec = embed([q])[0]
top = search(q_vec, top_k=1)
print(top.text)
```

你现在不用实现 embed/search 的细节，你只要理解：

- 向量库做的是“相似度搜索”
- 它返回的是“原文片段”，你再把这些片段塞进 Prompt

## 6. RAG 结果如何注入 Prompt（你必须会讲）

正确做法通常是：

- 检索得到 top_k 片段
- 拼成“参考资料”字段
- 在 Prompt 里明确告诉模型：
  - “只能基于 PRD + 参考资料生成”
  - “若资料不足请输出缺失点/假设”

你需要在 `test_point_extractor.py` / `test_case_generator.py` 找到调用 `rag_service` 的位置，理解它是如何把片段合并进 prompt 的。

## 7. 从 0-1 写一个最小 RAG（新手可复刻）

建议顺序：

### 7.1 先实现文档解析

- PDF/DOCX/MD/TXT → 纯文本

### 7.2 再实现分块

- 先做最简单的固定长度 + overlap

### 7.3 再实现 embedding

- 先用本地 embedding（例如 sentence-transformers）保证可离线

### 7.4 再实现向量库

- 先选一个本地持久化的向量库（Chroma 很适合原型）

### 7.5 最后接入生成流程

- 在生成前检索相关片段
- 拼进 Prompt

为什么这个顺序？

- 每一步都可单独验证
- 你能定位问题在“解析/分块/embedding/检索/注入”哪一环
