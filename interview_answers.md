# CaseGen 项目面试题参考答案

---

## 1. 介绍一下你的这个项目，包括业务形态，业务架构，业务流程

CaseGen 是一个 **AI 驱动的测试用例自动生成平台**，核心目标是将传统人工从 PRD 文档中提取测试点、编写测试用例的工作自动化，大幅提升 QA 团队的效率。

**业务形态：** 这是一个 B/S 架构的内部工具平台，面向 QA 工程师和测试团队使用。用户上传 PRD 文档（支持 PDF/DOCX/Markdown），平台通过大模型自动提取测试点并生成可执行的测试用例，最终导出为 Excel 供团队使用。同时支持知识库功能，利用 RAG 技术引入历史用例、业务规则等私域知识来提升生成质量。

**业务架构：**

- **前端：** Vue 3 + TypeScript + Element Plus，提供项目管理、文档上传、测试点/用例的查看编辑、知识库管理、AI 设置等功能
- **后端：** FastAPI（异步框架）+ SQLAlchemy 2.0（async ORM）+ SQLite
- **AI 层：** LiteLLM 统一对接多种大模型（GPT-4o、Claude、DeepSeek 等），支持灵活切换
- **RAG 层：** ChromaDB 向量数据库 + sentence-transformers 本地 Embedding，用于知识库的构建和检索
- **集成层：** 飞书 Webhook 通知，AI 任务完成后自动推送消息
- **部署：** Docker Compose 一键部署

**业务流程（核心链路）：**

```
创建项目 → 上传PRD文档 → 文档解析(PDF/DOCX→Markdown)
→ (可选)创建知识库并上传参考资料 → 知识库文档分块→Embedding→存入ChromaDB
→ AI提取测试点(LLM + RAG检索) → 人工Review/编辑测试点
→ AI生成测试用例(每个测试点生成1-3条用例) → 人工Review/编辑
→ 导出Excel → 飞书通知团队
```

整个 AI 任务采用**异步后台执行**模式：前端发起请求后立即返回 batch_id，后端通过 `asyncio.create_task` 后台执行 LLM 调用，前端轮询批次状态获取进度。

## 2. 对于用例生成的准确率和采纳率大概是多少，你觉得要提升，优化的方向还有哪些

根据实际使用情况，**测试点提取的准确率大约在 70%-80%**，**用例的采纳率大约在 40%-60%** 之间。如果配合知识库（RAG）引入业务规则和历史用例，采纳率能提升到 60% 左右。这个数据与业界（如百度的 40%-60%）基本一致。

主要的问题在于：

- 生成的用例偏通用化，缺少业务深度，特别是复杂业务逻辑的边界场景容易遗漏
- 对 PRD 中隐含的业务规则理解不够，会生成一些"看起来正确但实际不符合业务"的用例

**优化方向：**

1. **知识库质量提升：** 构建更高质量的私域知识库，包括历史优秀用例、业务规则文档、接口文档等，让 RAG 检索到更精准的参考信息
2. **Prompt 工程优化：** 引入更详细的约束框架，比如用例编号规则、特定的测试设计方法（等价类、边界值、判定表等），让输出更规范
3. **需求文档预处理：** 在 LLM 生成前增加一步"需求理解与澄清"，让模型先梳理功能模块和业务规则，再进行测试点提取
4. **人机协同反馈闭环：** 目前虽然支持重新生成（带 feedback），但还可以做得更细，比如对每个用例标记"采纳/拒绝/修改"，用这些反馈数据持续优化 Prompt
5. **分层生成策略：** 对于简单需求直接一步生成用例，复杂需求先拆解再分步生成，大文档先分块分别处理再汇总
6. **混合检索：** 现在只用向量相似度检索，可以加入关键词检索（BM25）做混合检索，提升召回率

## 3. RAG 是什么，谈谈你的理解，你觉得 RAG 的难点在哪里

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与大模型生成能力结合的技术范式。核心思路是：在调用 LLM 生成回答之前，先从外部知识库中检索与问题相关的信息片段，将这些片段作为上下文注入到 Prompt 中，让模型在回答时能参考最新的、领域特定的知识，从而减少幻觉、提升准确性。

**在我们项目中的应用：**

用户可以创建知识库，上传历史测试用例、业务规则文档、接口文档等。系统将这些文档分块（chunk_size=512 tokens），通过 sentence-transformers 生成 Embedding 向量，存入 ChromaDB。在 AI 提取测试点或生成用例时，用文档内容或测试点描述作为 query，从知识库中检索 top_k 个最相关的片段，注入到 Prompt 的上下文中。

**RAG 的难点：**

1. **分块策略（Chunking）：** 这是最基础也是最关键的问题。块太大会引入噪音、浪费 token 预算；块太小会丢失上下文语义。我们目前用固定字符分块（512 tokens + 64 tokens overlap），但理想情况下应该做语义分块（按段落、章节等自然边界切分）

2. **检索质量（Retrieval Quality）：** 向量相似度检索不等于语义相关性。比如用"登录功能"检索，可能召回"注册功能"的片段（因为向量相近），但实际不是用户需要的。解决方案包括混合检索（向量 + BM25 关键词）、重排序（Reranking）

3. **Embedding 模型选择：** 我们用的 all-MiniLM-L6-v2 是英文模型，对中文文档的语义理解能力有限。应该换用中文优化的 Embedding 模型（如 bge-base-zh、m3e 等）

4. **噪声过滤：** 检索到的片段不一定都有用，低质量的片段反而会干扰 LLM 的判断。需要设置相似度阈值过滤，或用 LLM 对检索结果做二次判断

5. **上下文窗口管理：** 检索到的片段需要和原始文档内容一起放入 Prompt，总长度不能超过模型的上下文窗口。需要合理分配 token 预算

## 4. PRD 文档不规范怎么解决

PRD 文档不规范是实际应用中最常见的问题，直接影响测试点提取的质量。我们从几个层面来解决：

**1. 文档解析层面的容错：**

- 支持多格式解析（PDF 用 pymupdf4llm、DOCX 用 python-docx、Markdown 直接读取），每种格式都有针对性的解析逻辑
- PDF 解析会保留原始的标题层级、表格结构，转为 Markdown 后结构更清晰
- 解析状态跟踪（pending → completed / failed），失败时可以重新解析

**2. Prompt 层面的兜底：**

- 在 System Prompt 中要求 LLM "仔细阅读提供的 PRD 文档"，即使文档结构混乱，LLM 也有一定的理解和整理能力
- 要求输出必须包含 source_context（来源引用），可以追溯每个测试点对应的 PRD 片段，方便人工验证

**3. 知识库补充：**

- 如果 PRD 写得不清楚，可以在知识库中补充业务规则文档、接口文档、设计文档等，通过 RAG 检索补充上下文信息，弥补 PRD 的不足

**4. 人工 Review + 反馈重新生成：**

- 系统支持对提取的测试点进行人工编辑和删除
- 支持"重新生成"功能，用户可以输入 feedback（比如"缺少异常场景"、"需要覆盖 XX 业务规则"），系统会结合已有测试点和用户反馈重新生成

**5. 流程层面（如果推广的话）：**

- 可以制定 PRD 模板规范，要求产品经理按统一模板编写
- 在上传前增加文档质量检测（比如检查是否有功能描述、是否有流程图等），给出质量评分和改进建议

## 5. 实现平台遇到了哪些难解决的问题，最后怎么解决的

**问题 1：LLM 输出的 JSON 格式不稳定**

LLM 返回的 JSON 经常出现各种问题：被 Markdown 代码块包裹、被截断（超过 max_tokens）、字段名不一致（test_points vs testPoints vs points）。

**解决方案：** 在 `llm_client.py` 中实现了多层 JSON 解析容错：

- 自动去除 Markdown 代码块包裹
- 截断修复：找到最后一个完整的 JSON 对象，补齐缺失的括号
- 在 `test_point_extractor.py` 中用 `_extract_points_list()` 函数兼容多种 key 名（test_points、testPoints、points、data、results）

**问题 2：异步任务管理和状态一致性**

AI 调用耗时长（几十秒到几分钟），如果用同步请求会超时。但异步执行后，如何保证数据库状态一致性是个难题——比如生成过程中服务重启，batch 会永远停在 running 状态。

**解决方案：**

- 使用 `asyncio.create_task` 后台执行，API 立即返回 batch_id
- 每个 batch 记录 started_at、completed_at，可以根据时间判断是否超时
- 异常时在 except 中更新 batch 状态为 failed 并记录 error_message
- 测试点在生成期间标记为 `status: generating`，完成后恢复为 `active`，即使出错也会恢复

**问题 3：不同 LLM 模型的兼容性**

不同模型对 JSON mode、temperature 参数的支持不一致，比如 GPT-5 推理模型只支持 temperature=1。

**解决方案：** 在 LLMClient 中实现了 `_resolve_temperature()` 方法，根据模型名称自动选择合适的 temperature。通过 LiteLLM 统一接口抹平不同供应商的差异。

**问题 4：ChromaDB 同步 API 与 FastAPI 异步架构的冲突**

ChromaDB 只提供同步 API，直接在 async 函数中调用会阻塞事件循环。

**解决方案：** 所有 ChromaDB 操作都用 `loop.run_in_executor(None, lambda: ...)` 包装，放到线程池中执行，避免阻塞主事件循环。

**问题 5：API Key 安全管理**

用户通过前端设置 API Key，需要防止明文泄露和非法字符导致请求失败。

**解决方案：** API Key 输入时做 ASCII 校验和 whitespace trim，响应时做掩码处理（只显示前6后4位），同时支持环境变量和 settings.json 两种方式配置。

## 6. 遇到了很大的需求文档，解析很耗时怎么办

大文档（几十页甚至上百页的 PRD）带来两个问题：一是解析慢，二是超过 LLM 的上下文窗口。

**解析层面的优化：**

1. **异步解析：** 文档上传后，解析操作在后台异步执行（`_parse_and_update`），不阻塞用户操作。知识库文档的处理也是后台异步，用 `asyncio.create_task` 启动
2. **状态跟踪：** 每个文档有 parse_status（pending → parsing → completed/failed），前端可以实时看到解析进度
3. **选择高效的解析库：** PDF 用 pymupdf4llm，底层是 C 实现的 MuPDF，速度比纯 Python 库快很多

**LLM 调用层面的优化：**

4. **文档内容截断：** 在 RAG 检索时，query 只取文档前 2000 字符（`document_content[:2000]`），避免整篇文档做 embedding 检索
5. **合并多文档时有分隔符：** 多文档合并用 `---` 分隔，帮助 LLM 区分不同文档的内容

**未来可进一步优化的方向：**

6. **文档分段处理：** 大文档按章节/模块拆分，分别提取测试点后汇总，避免一次性输入过长内容
7. **Map-Reduce 模式：** 大文档先分块让 LLM 各自提取测试点（Map），再汇总去重合并（Reduce）
8. **流式处理：** 使用 LLM 的 streaming 模式，边生成边展示，提升用户体验
9. **缓存机制：** 对已解析的文档缓存 Markdown 结果（已实现，存在 parsed_markdown 字段），修改前不需要重新解析

## 7. 用例生成很耗时，你是怎么去优化的

用例生成耗时主要来自两方面：LLM 调用本身的延迟（每次调用 3-10 秒），以及多个测试点需要逐个调用 LLM。

**已实现的优化：**

1. **异步非阻塞架构：** 整个后端基于 FastAPI 异步框架，LLM 调用使用 `litellm.acompletion`（异步），不阻塞其他请求的处理

2. **后台任务 + 立即响应：** AI 任务通过 `asyncio.create_task` 在后台执行，API 立即返回 batch_id，用户不需要等待。前端轮询 batch 状态展示进度

3. **指数退避重试：** LLM 调用失败时用指数退避重试（2^n 秒，最多 3 次），避免因偶发网络问题导致整个任务失败

4. **JSON 模式：** 使用 `response_format: json_object` 让 LLM 直接输出结构化 JSON，减少后处理开销

5. **飞书通知：** 生成完成后自动发送飞书通知，用户不需要一直盯着页面等待

**可进一步优化的方向：**

6. **并发调用：** 目前多个测试点是串行逐个调用 LLM 的（for 循环），可以改为用 `asyncio.gather` 或信号量控制的并发调用，比如同时处理 3-5 个测试点

7. **批量生成：** 将多个相关的测试点合并到一次 LLM 调用中，减少网络往返次数（需要平衡 token 消耗和输出质量）

8. **缓存策略：** 对相似的测试点做缓存，避免重复生成

9. **模型选择优化：** 提供快速模式（用 GPT-4o-mini 等小模型快速生成）和精准模式（用 GPT-4o 等大模型精确生成），让用户根据场景选择

10. **SSE/WebSocket 推送：** 用服务端推送替代前端轮询，减少不必要的请求，同时支持流式展示生成过程

## 8. Skill 和 MCP 的区别在哪里，你觉得这个平台能集成 MCP 或者 Skill 吗

**Skill（技能）** 是 Claude Code 等 AI 助手中的概念，本质上是**预定义的 Prompt 模板 + 工具调用组合**。Skill 在 AI 助手内部执行，是一种"快捷指令"，把复杂的多步操作封装成一个命令。它的作用域是 AI 助手自身，扩展的是 AI 的能力边界。

**MCP（Model Context Protocol，模型上下文协议）** 是 Anthropic 提出的开放标准协议，定义了 AI 应用与外部工具/数据源之间的通信规范。MCP 采用 Client-Server 架构，AI 应用作为 Client，各种工具（数据库、API、文件系统等）作为 Server。它解决的是**互操作性**问题——让不同的 AI 模型都能以统一的方式调用外部工具。

**核心区别：**

| 维度 | Skill | MCP |
|------|-------|-----|
| 本质 | Prompt 模板 + 工具组合 | 标准化通信协议 |
| 作用域 | AI 助手内部 | AI 应用与外部系统之间 |
| 标准化 | 各家实现不同 | 开放标准，跨模型通用 |
| 扩展性 | 添加新 Prompt 模板 | 添加新的 MCP Server |

**能否集成到 CaseGen 平台：**

**可以，而且两者都有价值：**

- **集成 MCP：** CaseGen 可以作为一个 MCP Server，暴露"提取测试点"、"生成用例"、"查询知识库"等能力。这样任何支持 MCP 的 AI 助手（Claude、Cursor 等）都能直接调用 CaseGen 的能力，不需要打开 Web 界面。反过来，CaseGen 也可以作为 MCP Client，调用外部的 MCP Server（比如 Jira MCP Server 自动创建测试任务、Git MCP Server 读取代码变更）

- **集成 Skill：** 可以把常用的操作封装为 Skill，比如 `/prd-to-test` 一键从 PRD 生成用例、`/export-cases` 导出用例等。在 IDE 插件场景下特别有用

## 9. 还有什么其他办法来做用例生成，能更好地提升准确率

除了当前的"PRD → LLM 直接提取"方案，还有以下思路可以显著提升准确率：

**1. Agentic RAG（智能体驱动的 RAG）：**

不是简单地检索一次就生成，而是让 Agent 自主决定需要查什么、查几次。比如：Agent 先分析 PRD，识别出涉及"支付"模块，然后主动去知识库中搜索"支付相关的业务规则"和"历史支付测试用例"，再综合生成。如果生成结果的置信度低，Agent 会自动重新检索或换个角度查询。

**2. 需求原子化拆分（两步法）：**

先让 LLM 将复杂需求拆解为**不可再分的功能点/业务规则**，人工确认后，再基于每个原子需求生成测试用例。这样每次 LLM 的输入更聚焦，输出质量更高。百度的实践证明这种分层策略能有效提升采纳率。

**3. 测试设计方法驱动：**

在 Prompt 中显式要求 LLM 运用经典测试设计方法：

- 等价类划分（有效/无效输入）
- 边界值分析（最小值、最大值、临界值）
- 判定表/因果图（多条件组合）
- 状态迁移图（流程类功能）

这比笼统地要求"生成正向/反向/边界"更精准。

**4. 多 Agent 协作：**

设计多个专门化的 Agent：

- **需求分析 Agent：** 理解和拆解 PRD
- **测试设计 Agent：** 基于测试方法论提取测试点
- **用例编写 Agent：** 将测试点转化为详细用例
- **审查 Agent：** 检查用例的完整性和合理性，给出改进建议

每个 Agent 各司其职，通过协作提升整体质量。

**5. Few-Shot + 历史用例模板：**

从知识库中检索相似功能的历史优秀用例，作为 Few-Shot 示例放入 Prompt。让 LLM "照葫芦画瓢"，输出风格和深度与团队已有用例保持一致。

**6. Self-Reflection（自我反思）：**

生成用例后，让 LLM 自己评审一遍："这些用例是否覆盖了所有功能点？是否有遗漏的异常场景？"，然后输出改进版本。类似 Chain-of-Thought 但应用在验证环节。

**7. 代码变更驱动的增量生成：**

对接代码仓库，分析 code diff 影响到的功能模块，只针对变更部分生成/更新测试用例，而不是每次都从 PRD 全量生成。

## 10. 什么是向量数据库，原理是什么，你是怎么构建的

**向量数据库**是专门用于存储和检索高维向量数据的数据库。传统数据库用精确匹配查询（SQL WHERE），向量数据库用**相似度计算**查询——给定一个查询向量，找到与它最相似的 K 个向量。

**核心原理：**

1. **Embedding（向量化）：** 将文本通过 Embedding 模型转化为固定维度的浮点数向量（比如 384 维）。语义相近的文本，其向量在高维空间中距离也近

2. **相似度度量：** 常用余弦相似度（Cosine Similarity）或欧氏距离。我们项目中使用余弦距离（`hnsw:space: cosine`）

3. **近似最近邻搜索（ANN）：** 在百万级向量中做精确最近邻搜索太慢（O(n)），所以用 ANN 算法加速。ChromaDB 底层使用 HNSW（Hierarchical Navigable Small World）算法，通过构建多层图索引，将搜索复杂度降到 O(log n)

**在项目中的构建方式：**

我们使用 **ChromaDB**（PersistentClient 模式，数据持久化到本地磁盘 `chromadb_data/` 目录）：

```
知识库文档上传 → 文档解析为文本 → 文本分块(chunk_size=512 tokens, overlap=64 tokens)
→ 每个 chunk 用 sentence-transformers(all-MiniLM-L6-v2) 生成 384维 Embedding
→ 存入 ChromaDB Collection(每个知识库一个 Collection，命名 kb_{uuid})
→ 同时存储元数据(source_filename, chunk_index, kb_document_id)
```

**检索流程：**

```
用户查询(PRD内容或测试点描述) → 生成查询 Embedding
→ 在目标 Collection 中做余弦相似度搜索 → 返回 top_k(默认5) 个最相关片段
→ 按相似度分数排序 → 格式化为 RAG 上下文注入 Prompt
```

**关键设计决策：**

- 选择 ChromaDB 而非 Milvus/Pinecone，因为它零配置、文件级持久化，适合单机部署场景
- Embedding 默认用本地模型（无需 API 调用），也支持切换 OpenAI Embedding
- 每个知识库独立 Collection，删除知识库时可以干净地删除对应的 Collection

## 11. 用例生成的 Prompt 是怎么设计的

Prompt 设计采用 **System Prompt + User Prompt 分离**的模式，集中管理在 `prompts.py` 中。

**测试点提取的 Prompt 设计：**

**System Prompt** 定义角色和规则：

- **角色设定：** "你是一位资深 QA 测试专家"——赋予模型专业身份
- **任务定义：** 明确要求从 PRD 中提取可测试的功能点和场景
- **覆盖性要求：** 6 条具体规则，包括覆盖所有功能模块、包含正向/反向/边界/异常、考虑非功能性测试（安全/性能/兼容）
- **优先级标准：** 定义 P0-P3 的含义（P0 核心流程、P1 重要功能、P2 一般功能、P3 边缘场景）
- **分类体系：** functional / edge_case / performance / security
- **输出格式约束：** 严格定义 JSON Schema，包含 title、description、priority、category、preconditions、expected_result、source_context 七个字段，并给出示例

**User Prompt** 提供具体内容：

- 注入解析后的文档内容（Markdown 格式）
- 如果有 RAG 上下文，通过 `RAG_CONTEXT_BLOCK` 模板注入检索到的知识库片段，格式为 `[来源: 文件名] + 文本内容`

**测试用例生成的 Prompt 设计：**

**System Prompt：**

- 角色转换为"资深 QA 测试工程师"
- 要求每个测试点生成 1-3 条用例（覆盖 positive/negative/boundary/edge）
- 强调步骤必须"详细、可操作、新人也能按步骤执行"
- 定义用例的 JSON 格式：steps 数组中每步包含 step_number、action、expected_result

**User Prompt：**

- 注入单个测试点的完整信息（title、description、priority、category、preconditions、expected_result）
- 可选注入 RAG 上下文

**设计思路总结：**

1. **结构化约束：** 用 JSON Schema 严格定义输出格式，配合 `json_mode=True`，确保输出可解析
2. **角色驱动：** 赋予专业角色身份，引导模型从 QA 视角思考
3. **显式列举：** 把覆盖范围（正向/反向/边界/异常/安全/性能）显式列出，防止遗漏
4. **上下文注入：** 通过模板变量动态注入文档内容和 RAG 检索结果
5. **可迭代：** 支持重新生成时注入已有测试点和用户反馈，形成"生成-反馈-优化"循环
