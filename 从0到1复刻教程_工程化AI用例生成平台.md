 # 从 0 到 1 复刻教程：工程化 AI 用例生成平台（按当前项目现状）
 
 本文是“只看这份文档就能复刻一个同类项目”的教程。它不会要求读者去翻你原项目源码才能理解，但会把你的项目里已经固化的规则、表设计、接口契约、任务机制、RAG 策略讲到足够具体，照着做就能做出一个“项目结构与能力边界高度相似”的系统。
 
 你在这个项目里的核心贡献，不是“手写了多少行代码”，而是把系统做成了现在这种形态：可运行、可解释、可追踪、可扩展。AI 更适合承担重复和样板部分；你负责定义规则、边界、验收标准，并最终把链路跑通。
 
 ---
 
 ## 0. 你到底“定了什么”？（先给结论）
 
 你定了四类东西，最终让项目变成现在这样：
 
 1) **业务链路与用户体验规则**
 - 输入是什么（PRD/文档、知识库文档）
 - 产物是什么（测试点、测试用例）
 - AI 操作如何交付给用户（异步批次 + 轮询状态 + 可关闭弹窗继续跑）
 
 2) **数据模型（表）与字段语义**
 - Project / Document / TestPoint / TestCase / GenerationBatch / KnowledgeBase / KnowledgeBaseDocument / Notification
 - 每张表的字段、默认值、状态枚举、关联关系
 
 3) **接口契约（API 协议）**
 - AI 触发接口返回 batch_id（而不是同步结果）
 - 统一的 batch 状态查询接口
 - 请求 DTO 与响应 DTO（字段名固定，前端按此实现）
 
 4) **三大工程化机制（你最该在面试里讲的“想法”）**
 - 异步批次：先返回 batch_id，后台跑任务，前端轮询状态
 - 结构化输出：强制 JSON 输出 + 解析兜底 + 落库前校验
 - 可观测可追踪：记录状态、错误原因、耗时与 token 消耗
 
 你让 AI 做的，主要是：
 - CRUD/分页/列表/表单这类重复实现
 - Pydantic schema/前端页面组件这类样板
 - 但前提是你把上面 4 类“硬约束”写清楚，并对 AI 产出做 review 与验收

 ## 1. 最小可运行目标（复刻时的验收标准）
 
 复刻到“最小可运行”时，你应该能完成下面 7 条验收：
 
 1) 后端服务启动成功，GET /health 返回 {"status":"ok"}
 2) 创建项目：能 POST 创建 Project，并列表查询
 3) 上传文档：能上传 pdf/docx/md，保存到磁盘，并写入 Document 记录
 4) 文档解析：Document.parse_status 从 pending → parsing → completed（失败则 failed + parse_error）
 5) 触发 AI 提取测试点：立即返回 batch_id，批次状态可查询
 6) 批次完成后：TestPoint 落库可查询（至少 title/description 有值）
 7) 前端可选：弹出“AI 生成中”进度弹窗，轮询 batch 状态；关闭弹窗任务继续，完成后触发刷新
 
 扩展到 RAG 后，再额外满足：
 - 创建知识库 + 上传知识库文档（pdf/docx/md/txt）
 - 知识库文档后台处理：status 从 pending → processing → completed（失败则 failed + error_message）
 - AI 生成时可传 knowledge_base_ids，生成流程会检索 topK 片段拼进 prompt

 ## 2. 项目骨架（目录与模块边界，按你现状）
 
 推荐的目录骨架（与你项目一致的分层）：
 
 - backend/app/main.py：FastAPI 入口、CORS、lifespan 建表、健康检查
 - backend/app/database.py：异步引擎与 Session、get_db 依赖注入
 - backend/app/api/v1/router.py：统一挂载 /api/v1 下的业务路由
 - backend/app/api/v1/*.py：按业务域拆路由（projects/documents/test_points/test_cases/ai_operations/knowledge_bases）
 - backend/app/models/*.py：SQLAlchemy ORM 表模型
 - backend/app/schemas/*.py：Pydantic 请求/响应 DTO
 - backend/app/services/*.py：业务服务（文档、知识库、通知等）
 - backend/app/services/ai/*.py：AI 相关服务（LLMClient、prompt、提取/生成管道、文档解析）
 - backend/app/services/rag_service.py：RAG 检索与索引
 - frontend/src/views：页面（项目列表/项目详情/设置）
 - frontend/src/components：通用组件（进度弹窗等）
 - frontend/src/api：前端 API 封装（axios）
 
 你定的分层规则只有一句话：
 
 - 路由层：只做参数与依赖注入、调用 service、返回 DTO
 - service 层：承载完整业务流程（尤其是 AI 任务流水线）
 - model 层：只描述表结构与关系
 - schema 层：只描述对外协议（请求/响应），不强依赖 ORM

 ## 3. 表结构怎么“按你项目现状”制定（每张表的字段、默认值、状态枚举）
 
 本节把你当前项目已有的表完全按现状列出来。复刻者照着建表即可。
 
 ### 3.1 公共约定（所有表共享）
 
 - 主键 id：String(36)，默认生成 UUID
 - created_at / updated_at：自动时间戳
 
 ### 3.2 projects（项目容器）
 
 表名：projects
 
 字段：
 - id：String(36)，PK，UUID
 - name：String(255)，非空
 - description：Text，可空
 
 关系（语义）：
 - 一个项目包含 documents/test_points/test_cases/generation_batches/notifications/knowledge_bases
 
 你在这里定的规则：
 - “项目”是所有数据的根容器，后续一切查询都能以 project_id 作为隔离边界
 
 ### 3.3 documents（输入文档）
 
 表名：documents
 
 字段：
 - id：String(36)，PK
 - project_id：String(36)，FK(projects.id)，非空
 - filename：String(255)，非空
 - file_type：String(20)，非空（实际值示例：pdf/docx/md）
 - file_path：String(512)，非空（磁盘路径）
 - file_size：Integer，可空
 - raw_text：Text，可空（解析得到的纯文本）
 - parsed_markdown：Text，可空（解析得到的 markdown）
 - parse_status：String(20)，默认 pending
 - parse_error：Text，可空
 
 解析状态枚举（按你现状）：
 - pending：刚上传
 - parsing：解析中
 - completed：解析成功
 - failed：解析失败（parse_error 有值）
 
 你在这里定的规则：
 - 文档不是“只存文件”，还要存解析产物（raw_text/markdown），否则后续 AI 生成无法稳定复现
 - 文档解析要可追踪：parse_status + parse_error
 
 ### 3.4 test_points（测试点，中间产物）
 
 表名：test_points
 
 字段：
 - id：String(36)，PK
 - project_id：String(36)，FK(projects.id)，非空
 - document_id：String(36)，FK(documents.id)，可空（可追溯来源）
 - title：String(500)，非空
 - description：Text，非空
 - priority：String(20)，可空（实际默认由 AI 给，或用 P2）
 - category：String(100)，可空（functional/edge_case/performance/security）
 - preconditions：Text，可空
 - expected_result：Text，可空
 - source_context：Text，可空（从 PRD 引用的片段）
 - is_manual_edit：bool，默认 False
 - generation_batch_id：String(36)，可空（追踪来源批次）
 - status：String(20)，默认 active（生成用例时会临时改为 generating）
 - sort_order：Integer，默认 0（前端展示排序）
 
 status 取值（按你现状）：
 - active：正常
 - generating：生成用例中（用例生成完成后恢复 active）
 
 你在这里定的规则：
 - 测试点必须“可编辑、可落库、可排序、可追踪来源批次”，所以需要 generation_batch_id、is_manual_edit、sort_order、status
 - title/description 必须非空：这是“落库前校验”的硬约束
 
 ### 3.5 test_cases（测试用例，最终产物）
 
 表名：test_cases
 
 字段：
 - id：String(36)，PK
 - project_id：String(36)，FK(projects.id)，非空
 - test_point_id：String(36)，FK(test_points.id)，非空
 - title：String(500)，非空
 - preconditions：Text，可空
 - steps：JSON，非空（列表，元素包含 step_number/action/expected_result）
 - priority：String(20)，可空
 - case_type：String(50)，可空（positive/negative/boundary/edge）
 - status：String(20)，默认 draft
 - generation_batch_id：String(36)，可空
 
 你在这里定的规则：
 - steps 用 JSON 列存，而不是拆成 steps 表：便于快速迭代与导出（面试里可以说“原型期更快，上生产可做规范化拆表”）
 - 用例也要可追踪来源批次：generation_batch_id
 
 ### 3.6 generation_batches（AI 批次状态机）
 
 表名：generation_batches
 
 字段：
 - id：String(36)，PK
 - project_id：String(36)，FK(projects.id)，非空
 - batch_type：String(30)，非空（示例：test_point_extraction / test_case_generation）
 - document_ids：JSON，可空（测试点提取时记录 document_ids）
 - llm_provider：String(50)，可空（记录选择的模型/提供商标识）
 - token_usage：JSON，可空（记录 prompt/completion/total）
 - status：String(20)，默认 running（创建时就设为 running）
 - error_message：Text，可空
 - started_at：DateTime，可空
 - completed_at：DateTime，可空
 
 status 枚举（按你现状）：
 - running
 - completed
 - failed
 
 你在这里定的规则（可观测的核心）：
 - 每次 AI 生成必须落一条批次记录，保证：可查状态、可查错误、可统计成本、可统计耗时
 - AI 触发接口不返回结果，只返回 batch_id（批次机制是产品交付方式）
 
 ### 3.7 knowledge_bases / knowledge_base_documents（RAG 数据域）
 
 表名：knowledge_bases
 
 字段：
 - id：String(36)，PK
 - project_id：String(36)，FK(projects.id)，非空
 - name：String(255)，非空
 - description：Text，可空
 - document_count：Integer，默认 0（聚合）
 - chunk_count：Integer，默认 0（聚合）
 - status：String(20)，默认 active（active/processing/error）
 
 表名：knowledge_base_documents
 
 字段：
 - id：String(36)，PK
 - knowledge_base_id：String(36)，FK(knowledge_bases.id)，非空
 - filename：String(255)，非空
 - file_type：String(20)，非空（pdf/docx/md/txt）
 - file_path：String(512)，非空
 - file_size：Integer，可空
 - chunk_count：Integer，默认 0
 - status：String(20)，默认 pending（pending/processing/completed/failed）
 - error_message：Text，可空
 
 你在这里定的规则（RAG 不只是“一个开关”）：
 - 知识库是一个独立域：要有 KB、KB 文档、异步处理状态、分块统计
 - KB 文档处理也是后台任务：状态必须可追踪
 
 ### 3.8 notifications（通知记录）
 
 表名：notifications
 
 字段：
 - id：String(36)，PK
 - project_id：String(36)，FK(projects.id)，非空
 - event_type：String(50)，非空（示例：test_points_generated / test_cases_generated）
 - channel：String(20)，默认 feishu
 - payload：JSON，可空
 - status：String(20)，默认 pending
 - error_message：Text，可空
 - sent_at：DateTime，可空
 
 你在这里定的规则：
 - 通知也要可追踪：发没发出去、失败原因是什么

 ## 4. 规则怎么“按你现状”制定（每条规则都落到具体机制）
 
 这一节把“你定的规则”写成可执行规范。复刻者按规范写代码即可。
 
 ### 4.1 规则 R1：所有 AI 生成都是“异步批次交付”
 
 你制定的规范（执行层面）：
 - 触发接口必须：
   - 创建 GenerationBatch（status=running，started_at=now）
   - commit 并返回 batch.id
   - 立刻启动后台任务（create_task）
 - 后台任务必须：
   - 使用独立数据库 session（不能复用请求的 session）
   - 成功：batch.status=completed，写 token_usage，写 completed_at，commit
   - 失败：batch.status=failed，写 error_message，写 completed_at，commit
 - 对外查询必须：
   - 提供 GET /ai/batches/{batch_id} 返回 batch 状态（DTO 固定）
 
 你当前项目就是这样落地的（复刻者照抄模板）：
 - 测试点提取：创建 batch → create_task → _run_extraction → 成功/失败回写
 - 测试用例生成：创建 batch → 标记测试点 generating → create_task → _run_generation → 回写 + 恢复测试点状态
 
 ### 4.2 规则 R2：LLM 输出必须结构化（JSON），并有解析兜底
 
 你制定的规范：
 - 每个 AI 任务都必须定义“输出协议”（JSON 对象 + 固定 key）：
   - 测试点：{"test_points":[{title,description,priority,category,preconditions,expected_result,source_context}]}
   - 用例：{"test_cases":[{title,preconditions,steps:[{step_number,action,expected_result}],priority,case_type}]}
 - LLM 调用必须开启 json_mode（强制 response_format=json_object）
 - JSON 解析必须具备兜底：
   - 支持 markdown code block 包裹
   - 截断修复（尽量保留到最后一个完整条目）
 - 落库前必须校验：
   - 不是 dict 的条目丢弃
   - title/description 为空丢弃（测试点）
   - steps 结构要格式化（用例）
 
 ### 4.3 规则 R3：可观测可追踪是第一等公民
 
 你制定的规范：
 - 所有任务必须能回答：
   - 现在状态是什么？running/completed/failed
   - 失败原因是什么？error_message
   - 花了多少 token？token_usage
   - 花了多久？started_at/completed_at
 - 前端必须把这些信息展示出来（至少：状态、错误、token 总数）
 - 用户体验规则：
   - 弹窗可以关闭，但任务继续；任务结束后仍要触发刷新（通过事件或轮询）
 
 ### 4.4 规则 R4：文档与知识库处理也要“状态可追踪”
 
 你制定的规范：
 - 文档上传后要解析，并把解析状态与错误落库（Document.parse_status/parse_error）
 - 知识库文档上传后走后台处理（pending→processing→completed/failed）
 - 成功后要更新聚合计数（KnowledgeBase.document_count/chunk_count）

 ## 6. 复刻实现步骤（你这套流程，怎么把活分给 AI 做）
 
 这一节直接回答：“我自己做什么？我让 AI 做什么？要给 AI 什么提示词？”
 
 ### 6.1 你自己做（必须由你制定的部分）
 
 复刻时你先写一页“规格说明”（不用长，关键是明确约束），包含：
 
 1) 实体与表（字段 + 状态枚举 + 关系）
 2) AI 任务交付方式（异步批次模板）
 3) 输出协议（JSON key 与字段清单）
 4) 失败路径（哪些失败要变成 batch.failed + error_message）
 5) RAG 策略（可选）：collection 命名、chunk 参数、topK、context 注入格式、query 构造方式
 6) 最小验收用例（第 1 节那 7 条）
 
 这些写清楚后，再把“样板与重复实现”交给 AI。
 
 ### 6.2 你让 AI 做（适合生成的部分）
 
 你可以让 AI 生成：
 - API 路由层的 CRUD（projects/documents/test_points/test_cases/knowledge_bases）
 - Pydantic schemas（请求/响应 DTO）
 - 前端页面表格、表单、弹窗（Element Plus）
 - 基础分页、排序、筛选的样板代码
 
 但你必须 review 的点：
 - 事务边界：commit/refresh 是否正确
 - 异步任务：是否用了独立 session
 - 错误处理：失败是否写入 batch/error_message
 - JSON 解析：是否严格按输出协议落库
 - RAG：是否按你定的策略（query/topK/context block）接入
 
 ### 6.3 你给 AI 的提示词模板（可直接复制使用）
 
 #### 模板 A：生成“异步批次 AI 任务”代码（后端）
 
 ```text
 你在一个 FastAPI + Async SQLAlchemy 项目里新增一个 AI 任务，必须严格遵循以下工程规范：
 
 1) 触发函数（service 层）必须：
    - 创建 GenerationBatch 记录：status="running"，batch_type=固定字符串，started_at=当前时间
    - 如果有输入 id 列表，写入 batch.document_ids 或等价字段（JSON）
    - db.add(batch) 后 commit，并 refresh(batch)
    - 立刻 asyncio.create_task 启动后台任务 _run_xxx(batch.id, ...)
    - 返回 batch（由 API 层序列化成 BatchStatusResponse）
 
 2) 后台任务 _run_xxx 必须：
    - 使用 async_session() 创建独立 session（不能复用请求的 db）
    - try 成功：写入结果表；batch.status="completed"；batch.token_usage=usage；batch.completed_at=now；commit
    - except 失败：batch.status="failed"；batch.error_message=str(e)；batch.completed_at=now；commit
 
 3) LLM 调用必须走统一客户端 LLMClient.chat(system_prompt,user_prompt,json_mode=True)
 
 4) 输出必须是 JSON object，并包含固定 key（例如 "test_points" 或 "test_cases"），解析后落库前要校验必填字段
 
 约束：不引入新依赖；保持现有命名风格；不要改动已存在的 BatchStatusResponse 字段名。
 输出：需要新增/修改的文件与关键代码块。
 ```
 
 #### 模板 B：生成 CRUD 路由 + Service（后端）
 
 ```text
 请为以下表生成 FastAPI 路由与 service（AsyncSession），包含 create/list/get/update/delete（按需要裁剪）：
 - 表字段与约束如下：（在这里粘贴你第 3 节的表定义）
 
 要求：
 - list 返回 items + total
 - get 不存在抛 NotFound
 - delete 返回 204
 - 更新只更新非空字段
 - 不要把大文本字段放到列表接口里（如 Document.raw_text/parsed_markdown）
 - 代码风格与已有项目一致（Pydantic v2 model_validate/from_attributes）
 ```
 
 #### 模板 C：生成前端页面（Element Plus）+ 按契约调用接口
 
 ```text
 你在 Vue3 + TS + Element Plus 项目里生成一个页面/组件。
 
 输入：
 - 后端接口契约（URL、请求字段、响应字段）
 - 页面要展示的表格列、表单字段、按钮与禁用规则
 
 必须实现的交互：
 - 触发 AI 任务后，拿到 batch_id 打开进度弹窗
 - 进度弹窗每 2 秒轮询 GET /ai/batches/{id}
 - 弹窗关闭不停止轮询；任务完成后触发父组件刷新列表
 - 失败时弹出错误提示（优先显示后端 detail 或 error_message）
 
 约束：不引入新依赖；axios 封装统一放在 src/api；状态管理沿用现有写法。
 ```

 ## 7. RAG 增量怎么做（按你现状的“策略与规则”复刻）
 
 复刻 RAG 的关键不是写代码本身，而是把策略定清楚。按你项目现状，RAG 的策略是：
 
 1) 知识库 = 多个 collection（每个 kb_id 一个 collection）
 - collection 名称规则：把 kb_id 变成 "kb_{kb_id}"（并把 '-' 替换为 '_'）
 
 2) 切块策略（chunk_text）
 - chunk_size / chunk_overlap 是可配置的（你现状用“字符近似 token”的做法：4 字符≈1 token）
 
 3) 索引策略（index_document）
 - KB 文档上传后后台处理：解析 → 分块 → embedding → 写入向量库
 - 把 source_filename、kb_document_id、chunk_index 写进 metadata，方便后续展示来源
 
 4) 检索策略（retrieve）
 - 支持多个 knowledge_base_ids 同时检索
 - 各 KB 检索 topK，合并后按相似度排序，取总 topK
 
 5) 注入策略（prompt 拼接）
 - 把检索到的片段拼成一个 rag_context 块，插入用户提示词里
 - rag_context 块带来源文件名，便于解释“证据从哪来”
 
 6) query 构造策略（生成时怎么搜）
 - 测试点提取：用 document_content 的前 2000 字做 query（避免过长）
 - 用例生成：用 测试点 title + description 做 query（更精准）
 
 复刻者只要把以上 6 条策略落到实现里，就能得到与你项目“效果与形态一致”的 RAG 接入。
